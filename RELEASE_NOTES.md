- **rao-007**: Phase 2 (Core Technical Architecture) completed and closed — new `docs/PHASE2-CORE-TECHNICAL-ARCHITECTURE.md` adds Service-Oriented Architecture conventions, a SOLID-principles mapping, Agent Engine internals (Registry/Lifecycle/Runner + concurrency model), a Workflow Engine design, a concrete Event Bus (built on `ActiveSupport::Notifications`), Conversation/Memory/Prompt architecture, and all ten cross-cutting engineering strategies. Three requirements carried forward to future implementation tasks: Event Bus subscribers must be non-blocking, the Concurrency Guard must be atomic (not check-then-act), and the dependency-graph cache needs insert+delete invalidation test coverage.
- **rao-008**: Phase 3 (Mock AI Provider Foundation) completed and closed — new `docs/PHASE3-MOCK-AI-PROVIDER-FOUNDATION.md` defines the Provider Interface (request/response/error/capability/configuration models), the Mock AI Provider's architecture and lifecycle, Provider-specific Conversation/Agent Execution flow detail, Prompt Management and the 11-category Prompt Template Library, the Mock Response Strategy (12 deterministic scenarios), Token/Cost Simulation, and the Future Migration Plan's mechanics. A pre-close-out revision pass caught and fixed a genuine correctness bug (multiple `tool_calls` from one turn would have collided on a shared idempotency key, silently dropping records) plus three completeness gaps (missing `memory_updates` field, non-round-aware fixture selection, under-specified `latency_ms`). One requirement carried forward: real providers must require validated, non-nil credentials before activation.
- **rao-009**: Phase 4 (Database Design) completed and closed — new `docs/PHASE4-DATABASE-DESIGN.md` deepens `docs/DATABASE-SCHEMA.md` with a Mermaid ERD, a consolidated Indexing Strategy, Constraints, an Enumerations catalog, a JSON Field Usage catalog, a State Machines catalog, a deepened Audit Tables mechanism, a Soft Delete Strategy, a Versioning Strategy, and Performance Considerations. Two new decisions: no soft deletes anywhere (one status-based exception for a deleted linked issue), and "one active prompt version per key" enforced at the application layer rather than a DB constraint (MySQL doesn't portably support partial unique indexes). Three requirements carried forward: a concurrent prompt-activation race-condition test, a `to_prepare` core-association boot-mode test, and a retention job that excludes non-terminal agent runs.

- **rao-016 implemented (untested)**: 18 `db/migrate/` migrations giving Phase 10's models their first real backing schema — one `create_table` per table in `docs/DATABASE-SCHEMA.md`, every index from `docs/PHASE4-DATABASE-DESIGN.md` §4, every FK declared at the DB level per §5 (deliberately no partial unique index for `prompt_templates.is_active` — MySQL portability). Status stays `specification`, not `done` — untested against a live Redmine instance. Also fixed a genuine FK-ordering gap in the ticket's own table list (`agent_runs` referenced `conversations` before that table existed in the listed order) by creating `conversations` before `agent_runs` on disk — logged as a transparent Gate 1 revision pass. Phase 10 + Phase 11 together give the plugin a bootable schema for the first time.
